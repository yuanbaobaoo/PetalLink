# 04 · 数据模型与持久化

> SQLite schemaVersion=6。两张核心表：`sync_items`（最后成功同步基线）+ `transfer_queue`（九态持久化状态机 + CAS 乐观锁）。
> 外加磁盘层：xattr 占位符（2 键：`com.hwcloud.state` + `com.apple.FinderInfo`）、CloudTreeCache（云端树快照）、syncstate（启动 diff 冗余）。
> v6 新增两张表：`local_inode_map`（inode→fileId 映射，文件身份识别的核心，替代原 fileId xattr）+ `free_up_staging`（释放空间事务，替代原 `com.hwcloud.freeUpRelativePath` xattr）。详见 §6.5。
> Kotlin 重构建议用 Room 或 SQLDelight，**字段语义、状态机迁移表、CAS revision 机制必须完整保留**。

## 1. 数据库基础

| 项 | 值 |
|---|---|
| DB 文件 | `<Application Support>/<bundle id>/petal_link.db` |
| 引擎 | SQLite（rusqlite, bundled） |
| schema 版本 | `schemaVersion = 6` |
| 外键 | `PRAGMA foreign_keys = ON`（连接初始化时设置） |

### 迁移策略

所有迁移用 `unchecked_transaction` 包裹，整体原子：

- **新库**（schemaVersion 为 0 或不存在）：直接 `create_all()` 写入**终态 schema**（v6 全部表/列/索引一次性建好，含 v6 新增的 `local_inode_map` + `free_up_staging`），然后 user_version = 6。
- **旧库**：读取 user_version，逐级 `upgrade(v → v+1)`，每个 `upgrade_vX` 函数只负责自己的增量 ALTER + 数据回填，最终也走到终态。

> 这种「新库直达终态 + 旧库逐级 upgrade」的双路径，避免新装用户走完 v2→v6 全部历史步骤。
> v6 迁移（`upgrade_to_v6`）是**纯增量**：只 CREATE 两张新表，不回填历史 inode（启动后首次扫描自动填充），不删旧 xattr（向后兼容）。详见 §5.5。

## 2. sync_items 表（最后成功同步基线）

记录每个文件「最后一次成功同步」后的基线状态，作为 3-way diff 的第三方参照（local × cloud × db）。

### 2.1 完整建表 SQL

```sql
CREATE TABLE IF NOT EXISTS sync_items (
    file_id           TEXT    NOT NULL,
    local_path        TEXT    NOT NULL,
    parent_folder_id  TEXT,
    name              TEXT    NOT NULL,
    is_folder         INTEGER NOT NULL DEFAULT 0,
    size              INTEGER NOT NULL DEFAULT 0,
    local_size        INTEGER,
    sha256            TEXT,
    local_mtime       INTEGER,
    cloud_edited_time INTEGER,
    last_sync_time    INTEGER,
    status            INTEGER NOT NULL DEFAULT 0,
    error_message     TEXT,
    PRIMARY KEY (file_id, local_path)
);
```

复合主键 `PRIMARY KEY (file_id, local_path)` 支持同名文件在云端不同父目录、以及同一本地路径在不同 fileId 下的多版本共存。

### 2.2 字段定义（13 个）

| 字段 | 类型 | 说明 |
|---|---|---|
| `file_id` | TEXT NOT NULL | 云端 fileId（或 `pending:<rel_path>`，见 §12）；与 `local_path` 组成主键 |
| `local_path` | TEXT NOT NULL | 相对挂载根的规范 UTF-8 路径；与 `file_id` 组成主键 |
| `parent_folder_id` | TEXT | 云端父目录 ID |
| `name` | TEXT NOT NULL | 文件名（不含路径） |
| `is_folder` | INTEGER NOT NULL DEFAULT 0 | 是否目录（0/1） |
| `size` | INTEGER NOT NULL DEFAULT 0 | 云端大小（字节） |
| `local_size` | INTEGER | 本地文件大小（字节） |
| `sha256` | TEXT | 本地文件 sha256（hex） |
| `local_mtime` | INTEGER | 本地 mtime（毫秒 epoch） |
| `cloud_edited_time` | INTEGER | 云端 editedTime（毫秒 epoch） |
| `last_sync_time` | INTEGER | 最后同步时间（毫秒 epoch） |
| `status` | INTEGER NOT NULL DEFAULT 0 | 见下表 |
| `error_message` | TEXT | 错误信息（仅 FAILED/CONFLICT 时有值） |

### 2.3 status 枚举

| 值 | 名称 | 含义 |
|---|---|---|
| 0 | SYNCED | 已同步 |
| 1 | CLOUD_ONLY | 仅云端 |
| 2 | LOCAL_ONLY | 仅本地 |
| 3 | SYNCING | 同步中 |
| 4 | FAILED | 失败 |
| 5 | CONFLICT | 冲突 |
| 7 | DELETED | 已删除（tombstone，防云端重建） |

⚠️ **值 6 被刻意跳过**。`DELETED=7` 是 tombstone：当本地文件被删除、但云端仍存在时，写入 `status=7` 防止下一次 diff 又把它当作 CLOUD_ONLY 重新下载——记录"已确认删除"的意图。新增枚举值时务必避开 6 以保持向前兼容。

### 2.4 repository 方法（sync_items.rs）

| 方法 | 行为 |
|---|---|
| `find_by_file_id(file_id)` | 按 fileId 查询；**多条命中报错"拒绝歧义基线"**（diff 不允许在多行间猜测，必须唯一） |
| `load_all()` | 全量加载；**过滤 `.hwcloud_` 前缀的路径**（系统元数据文件不参与 diff） |
| `upsert(item)` | `INSERT OR REPLACE`（按主键覆盖；不区分新建/更新） |
| `delete_by_local_path(local_path)` | 按本地路径删除 |
| `reset_stale_statuses()` | 启动时把残留的 `SYNCING(3)` 重置为 `SYNCED(0)`（上次崩溃留下的中间态，回退到上次已知好状态） |

## 3. transfer_queue 表（v5 九态状态机）

持久化传输任务的完整生命周期上下文：断点续传、远端核验、网络恢复续跑、退避重试都依赖此表的持久性。

### 3.1 完整建表 SQL（v5 终态）

```sql
CREATE TABLE IF NOT EXISTS transfer_queue (
    -- v1 基础列
    id                          INTEGER PRIMARY KEY AUTOINCREMENT,
    direction                   INTEGER NOT NULL,
    file_id                     TEXT,
    local_path                  TEXT,
    name                        TEXT    NOT NULL,
    total_size                  INTEGER NOT NULL DEFAULT 0,
    transferred                 INTEGER NOT NULL DEFAULT 0,
    state                       INTEGER NOT NULL DEFAULT 0,
    error_message               TEXT,
    created_at                  INTEGER NOT NULL,
    finished_at                 INTEGER,
    -- v2 分片上传上下文
    server_id                   TEXT,
    upload_id                   TEXT,
    resume_offset               INTEGER NOT NULL DEFAULT 0,
    -- v4 华为 resume 会话
    session_url                 TEXT,
    -- v5 状态机 + 上下文
    relative_path               TEXT,
    parent_file_id              TEXT,
    operation                   INTEGER,
    source_mtime                INTEGER,
    source_size                 INTEGER,
    expected_cloud_edited_time  INTEGER,
    attempt_count               INTEGER NOT NULL DEFAULT 0,
    next_retry_at               INTEGER,
    error_kind                  INTEGER,
    remote_result_file_id       TEXT,
    state_revision              INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX IF NOT EXISTS idx_transfer_queue_state        ON transfer_queue(state);
CREATE INDEX IF NOT EXISTS idx_transfer_queue_direction    ON transfer_queue(direction);
CREATE INDEX IF NOT EXISTS idx_transfer_queue_file_id      ON transfer_queue(file_id);
CREATE INDEX IF NOT EXISTS idx_transfer_queue_relative     ON transfer_queue(relative_path);
CREATE INDEX IF NOT EXISTS idx_transfer_queue_next_retry   ON transfer_queue(next_retry_at);
```

### 3.2 字段版本标注（25 字段 + 5 索引）

| 字段 | 类型 | 版本 | 说明 |

### 3.3 direction 枚举

| 值 | 名称 | 含义 |

| 0 | UPLOAD | 上传 |
| 1 | DOWNLOAD | 下载 |
| 2 | DELETE | 删除 |

### 3.4 operation 枚举（8 种，0–7）

| 值 | 名称 | 含义 |
|---|---|---|
| 0 | CREATE | 新建上传 |
| 1 | UPDATE | 覆盖上传 |
| 2 | DOWNLOAD | 下载 |

### 3.5 state 枚举（九态状态机）

| 值 | 名称 | 含义 |
|---|---|---|
| 0 | Pending | 等待执行 |
| 1 | Running | 正在执行 |
| 2 | WaitingForNetwork | 等待网络恢复 |
| 3 | BackingOff | 退避中（等待 next_retry_at 到来） |
| 4 | VerifyingRemote | 核验远端有歧义结果 |
| 5 | RestartRequired | 需重新规划（不能原样重试，需回 planner） |
| 6 | Completed | 已完成（**终态**） |
| 7 | Failed | 永久失败（**终态**） |
| 8 | Canceled | 已取消（**终态**） |

### 3.6 error_kind 枚举（12 种，0–11）

| 值 | 名称 |
|---|---|
| 0 | Network |
| 1 | Timeout |
| 2 | Auth |
| 3 | RateLimit |
| 4 | Server |
| 5 | Quota |
| 6 | Permission |
| 7 | Validation |
| 8 | SessionExpired |
| 9 | RemoteAmbiguous |
| 10 | LocalChanged |
| 11 | Unknown |

> 任何迁移前的错误归一化都会把 NULL 或无法识别的错误归为 `Unknown(11)`（见 §5 v5 迁移）。

### 3.7 合法状态迁移（can_transition）

```
Pending            → Running | WaitingForNetwork | RestartRequired | Failed | Canceled
Running            → WaitingForNetwork | BackingOff | VerifyingRemote | RestartRequired | Completed | Failed | Canceled
WaitingForNetwork  → Running | RestartRequired | Failed | Canceled
BackingOff         → Running | RestartRequired | Failed | Canceled
VerifyingRemote    → Running | WaitingForNetwork | BackingOff | RestartRequired | Completed | Failed | Canceled
RestartRequired    → Pending | VerifyingRemote | Failed | Canceled
Failed             → Pending | RestartRequired | Canceled
```

要点：
- **Completed(6) 与 Canceled(8) 是终态，无任何出边**（一旦达成不可再迁移）。
- 任何非法迁移抛出 `IllegalTransition` 错误。
- 所有迁移通过 CAS `state_revision` 乐观锁保护（见 §4）。

## 4. CAS revision 乐观锁（transfer_queue 核心）

多线程并发推进同一任务（如网络回调、退避定时器、用户取消同时发生）时，必须保证状态迁移串行化。本项目用 **`state_revision` 单调递增 + WHERE 条件比较** 的乐观锁，而非显式 mutex。

### 4.1 核心更新 SQL

```sql
UPDATE transfer_queue
SET    state = :new_state,
       state_revision = state_revision + 1,
       /* ...其它 patch 字段... */
WHERE  id = :id AND state_revision = :expected_revision;
```

`WHERE ... AND state_revision = ?` 是乐观锁的关键：调用方传入自己读取时看到的 revision，若期间已被他人改过则 affected rows = 0，迁移失败。

### 4.2 ColumnPatch 三态（针对 nullable 列）

更新某列时支持三种语义：

```rust
enum ColumnPatch<T> {
    Keep,           // mode 0: 保持原值不动（不出现在 SET 子句）
    Set(T),         // mode 1: 写入新值
    Clear,          // mode 2: 置 NULL
}
```

- **Keep(0)**：`SET` 子句不包含此列。
- **Set(1)**：`SET col = ?`。
- **Clear(2)**：`SET col = NULL`。

这样调用方可显式区分"不改"、"写新值"、"清空"三种意图，避免 `None` 语义模糊（`None` 既可能是"没传"也可能是"清空"）。

### 4.3 transition_transfer 完整流程

```
1. SELECT * FROM transfer_queue WHERE id = ?            -- 读当前 state + state_revision
2. 校验 expected_revision == row.state_revision         -- 否则并发已变，报错
3. can_transition(row.state → new_state)                -- 否则 IllegalTransition
4. UPDATE ... WHERE id = ? AND state_revision = ?
5. 检查 changed == 1                                    -- 否则报 StaleRevision
```

`changed != 1` 报 **`StaleRevision`**：在步骤 1 与步骤 4 之间有人抢先迁移过。调用方应重新 SELECT 再决策，而不是盲目重试。

### 4.4 transition_transfer_clearing_upload_session

特定迁移（进入 Failed / Canceled / RestartRequired 等需要丢弃上传会话的终态/中间态）会**在同一事务内原子地**：

- `server_id = NULL`
- `upload_id = NULL`

把分片上传会话一并失效，避免下次重试误用旧 session（华为云 session 过期再用会触发歧义错误）。

### 4.5 patch_transfer_in_state（不改生命周期）

仅更新错误/重试字段，**不触碰 state、不递增 state_revision**：

- `error_message`、`error_kind`、`attempt_count`、`next_retry_at`、`transferred`、`resume_offset` 等运行态字段。
- 用途：Running 中实时上报进度、记录瞬时错误但还不想触发状态迁移。

### 4.6 update_running_transfer（迟到回调守卫）

```sql
UPDATE transfer_queue
SET    transferred = ?, resume_offset = ?, ...
WHERE  id = ? AND state = 1;   -- 仅 Running(1) 时更新
```

**刻意不递增 `state_revision`**。原因：进度回调可能迟到，若它递增 revision，会让正在并发执行的状态迁移误判为 stale。把 WHERE 限定在 `state = Running` 即可——一旦任务迁出 Running，迟到的回调 affected rows = 0，自动作废，且不影响真正在推进的迁移的 revision。

### 4.7 prune_transfer_history(limit = 100)

保留最近 100 条**终态**记录（Completed / Failed / Canceled），更早的终态行删除，控制表大小。非终态任务**绝不裁剪**。

## 5. migrations 迁移链

每个 `upgrade_vN` 函数只处理 `v(N-1) → vN`，互不重叠。

### 5.1 v2：分片上传上下文

```sql
ALTER TABLE transfer_queue ADD COLUMN server_id     TEXT;
ALTER TABLE transfer_queue ADD COLUMN upload_id     TEXT;
ALTER TABLE transfer_queue ADD COLUMN resume_offset INTEGER NOT NULL DEFAULT 0;
```

### 5.2 v3：sync_items 加本地大小

```sql
ALTER TABLE sync_items ADD COLUMN local_size INTEGER;
```

### 5.3 v4：华为 resume 会话 URL

```sql
ALTER TABLE transfer_queue ADD COLUMN session_url TEXT;
```

### 5.4 v5：状态机上下文（完整）

**步骤 1：加 11 列**（用 `add_column_if_missing` 幂等加列）

```sql
ALTER TABLE transfer_queue ADD COLUMN relative_path              TEXT;
ALTER TABLE transfer_queue ADD COLUMN parent_file_id             TEXT;
ALTER TABLE transfer_queue ADD COLUMN operation                  INTEGER;
ALTER TABLE transfer_queue ADD COLUMN source_mtime               INTEGER;
ALTER TABLE transfer_queue ADD COLUMN source_size                 INTEGER;
ALTER TABLE transfer_queue ADD COLUMN expected_cloud_edited_time INTEGER;
ALTER TABLE transfer_queue ADD COLUMN attempt_count               INTEGER NOT NULL DEFAULT 0;
ALTER TABLE transfer_queue ADD COLUMN next_retry_at               INTEGER;
ALTER TABLE transfer_queue ADD COLUMN error_kind                  INTEGER;
ALTER TABLE transfer_queue ADD COLUMN remote_result_file_id       TEXT;
ALTER TABLE transfer_queue ADD COLUMN state_revision              INTEGER NOT NULL DEFAULT 0;
```

**步骤 2：`recover_legacy_relative_paths`** —— 从老 `local_path`（绝对路径）回填 `relative_path`（相对挂载根的稳定路径），保证升级后状态机的相对路径字段非空。

**步骤 3：错误归一化**（让老 Failed 行符合 v5 的 error_kind 约束）

```sql
UPDATE transfer_queue SET error_kind = 11  -- Unknown
WHERE state = 4 AND error_kind IS NULL;
```

> 这里 `state = 4` 是 **v4 之前的旧 state 编码**（旧编码中 4 = Failed）。

**步骤 4：state 编码迁移**（v4 旧 6 态 → v5 九态）

```sql
UPDATE transfer_queue SET state = CASE state
    WHEN 0 THEN 0   -- Pending → Pending
    WHEN 1 THEN 0   -- (旧 Running/入队) → Pending（重启后再调度）
    WHEN 2 THEN 0   -- (旧 queued)       → Pending
    WHEN 3 THEN 6   -- (旧 Done)         → Completed
    WHEN 4 THEN 7   -- (旧 Failed)       → Failed
    WHEN 5 THEN 8   -- (旧 Canceled)     → Canceled
END;
```

**步骤 5：无法安全恢复的活动任务标失败**

升级前处于 Running / 中间态的任务，重启后无法安全续跑（上下文不全），全部标 `state = Failed(7)` 且 `error_kind = Validation(7)`，由用户/planner 重新决策。

**幂等保证**：`add_column_if_missing` 通过 `PRAGMA table_info(...)` 查列名再决定是否 ALTER，避免重复迁移报"duplicate column"。

### 5.5 v6：inode 身份识别两张表（纯增量）

`upgrade_to_v6` **只建表，不动旧数据**——这是 inode 方案的关键设计（见 `11-基于inode的文件身份识别方案.md` §3.3）：

```sql
-- 文件身份映射（替代 com.hwcloud.fileId xattr）
CREATE TABLE IF NOT EXISTS local_inode_map (
    inode         INTEGER NOT NULL,
    relative_path TEXT    NOT NULL,
    file_id       TEXT    NOT NULL,
    scanned_at    INTEGER NOT NULL,
    PRIMARY KEY (inode)
);
CREATE INDEX IF NOT EXISTS idx_inode_map_path ON local_inode_map(relative_path);
CREATE INDEX IF NOT EXISTS idx_inode_map_fid  ON local_inode_map(file_id);

-- 释放空间事务（替代 com.hwcloud.freeUpRelativePath xattr）
CREATE TABLE IF NOT EXISTS free_up_staging (
    staging_name   TEXT    NOT NULL PRIMARY KEY,  -- 暂存文件名（如 .hwcloud_freeup-xxxx）
    relative_path  TEXT    NOT NULL,              -- 原始相对路径
    file_id        TEXT    NOT NULL,              -- 云端文件 ID
    source_mtime   INTEGER,                       -- 原文件 mtime（回滚恢复用）
    source_size    INTEGER,                       -- 原文件大小（回滚恢复用）
    created_at     INTEGER NOT NULL
);
```

迁移要点：
- **不回填历史 inode**：启动后首次扫描自动填充 `local_inode_map`。
- **不删除旧 xattr**：向后兼容，启动后由清理流程逐步移除（见 `11` §6.5）。
- 两张新表建好后 `user_version = 6`。
- 字段语义详见 §6.5。

## 6. xattr 占位符模型（macOS 专属）

为支持「按需下载」（Files-On-Demand-lite），云端文件在本地以**占位文件**形式存在：真实文件名、**0 字节空文件**，仅作目录结构占位与视觉标记，实际内容按需下载。

### 6.1 两个 xattr 键（inode 方案）

| xattr 键 | 取值 | 作用 |
|---|---|---|
| `com.hwcloud.state` | `placeholder` 或 `downloaded` | **占位状态的唯一权威判据** |
| `com.apple.FinderInfo` | 32 字节 xattr | Finder 灰标（label index 7） |

> `com.hwcloud.state` 是判别占位文件的**核心**：`placeholder` = 未下载，`downloaded` = 已下载。**xattr 是数据源头，文件大小/文件名/Finder 标签均不参与判定。**

> ⚠️ **inode 方案变更**（v6，见 `11-基于inode的文件身份识别方案.md`）：原 `com.hwcloud.fileId` / `com.hwcloud.size` / `com.hwcloud.freeUpRelativePath` 三个 xattr 键已删除。
> - **文件身份**（原 `com.hwcloud.fileId`）改由 `local_inode_map` 表（inode→fileId 映射）承担，详见 §6.5。身份识别不再依赖文件 xattr，从结构上消除"xattr 跟随复制导致同一身份多处出现"的 bug 根源。
> - **云端真实大小**（原 `com.hwcloud.size`）由 `stat` 直接读取，无需冗余存储。
> - **释放空间恢复路径**（原 `com.hwcloud.freeUpRelativePath`）改由 `free_up_staging` 表记录，详见 §6.5。

### 6.2 Finder 灰标实现（非 osascript）

直接读写 `com.apple.FinderInfo` xattr：

- **置灰标**：读出原 32 字节 buf → 修改 `buf[9] = 0x02`（即 label index 7 的灰标位）→ 写回（保留其它字段）。
- **清灰标**：把 `buf[9]` 清零 → 若整 buf 全零则 `remove_xattr`（彻底删除该 xattr），否则写回。

**不使用 `osascript` 调 Finder API**——直接 xattr 操作更快、无 GUI 依赖、可在无用户会话的守护进程上下文运行。

### 6.3 占位文件物理形态

- 用**真实文件名**（不是 `.downloading` 之类临时名）。
- **0 字节**——这是判定的辅助特征，但**不是权威判据**（权威是 `com.hwcloud.state`）。

### 6.4 状态流转 API

| API | 行为 |
|---|---|
| `create_placeholder_if_needed` | 创建占位（**带存在性检查**）：若已有文件且**无任何占位 xattr** → 视为用户文件，**绝不转换**（防误吞用户数据）；仅在缺失或已是占位时**写 state xattr + 灰标 + DB 写 inode 映射**（`local_inode_map` upsert） |
| `create_placeholder_strict` | **严格版**（破坏性流程专用）：**不做存在性检查，直接 `create_new`**——只写 state xattr + 灰标 + DB 写 inode 映射；调用方必须自己保证语义安全 |
| `mark_downloaded` | 内容写入完成后：`com.hwcloud.state` → `downloaded` + 清除 Finder 灰标 + **更新 inode 映射**（下载覆盖会换 inode，需 `upsertMapping` 新 inode，见 §6.5 下载覆盖兜底） |
| `backup_modified_placeholder_if_needed` | 检测到占位被用户写入内容（state 仍 `placeholder` 但 size > 0），下载云端新版本前先改名备份为 `原名.local-<timestamp>.ext`，并清掉备份的 state xattr。inode 方案下副本天然产生新 inode，自动被当新文件处理，无需额外撕身份 |
| `delete_local` | 删除本地文件：**0 字节且必须是 placeholder 才删**（保护 `.gitkeep` 等用户保留的空文件） |

> ⚠️ **Kotlin 重构注意**：xattr 是 macOS 专属。若 Kuikly 版本只做 Android 则此模型不适用；若仍做 macOS（Native/Kotlin-Native 或 JVM+JNI）则需保留 **2 个 xattr 键**（`com.hwcloud.state` + `com.apple.FinderInfo`）的读写语义。`com.apple.FinderInfo` 用 Foundation NSFileManager 或 libc `setxattr`/`getxattr` 均可。
>
> inode 读取：JVM 用 `Files.getAttribute(path, "unix:ino") as Long`；Kotlin-Native 用 POSIX `stat().st_ino`。macOS APFS/HFS+ 的 inode 是 `u64`，SQLite `INTEGER`（64 位有符号）足以容纳，Kotlin 侧用 `Long` ↔ inode 安全转换（inode 不会超过 `Long.MAX_VALUE`）。**本方案假设同步目录在本地卷**，外接/网络卷（SMB/NFS）inode 语义不稳定（见 `11` §6.4）。

### 6.5 inode 身份识别表与 identity 模块（v6 新增）

inode 方案把文件身份从磁盘 xattr 迁移到 DB，新增两张表（建表 SQL 见 §5.5）。完整设计见 `11-基于inode的文件身份识别方案.md`。

#### 6.5.1 `local_inode_map` 表（inode → fileId 映射）

身份识别的核心数据结构，独立于 `sync_items`，主键为 `inode`：

| 字段 | 类型 | 说明 |
|---|---|---|
| `inode` | INTEGER NOT NULL (PK) | 文件系统 inode 编号（`stat().st_ino`）；主键唯一保证同一 inode 不可能映射到多个 fileId |
| `relative_path` | TEXT NOT NULL | 相对挂载根路径，与 `sync_items.local_path` 同语义 |
| `file_id` | TEXT NOT NULL | 云端文件 ID，与 `sync_items.file_id` 对应 |
| `scanned_at` | INTEGER NOT NULL | 上次扫描到该 inode 的时间戳，用于清理陈旧记录 |

两个二级索引：`idx_inode_map_path`（按路径查）、`idx_inode_map_fid`（按 fileId 查）。

**为什么独立建表而非给 `sync_items` 加 `inode` 列**：
- 解耦：不是所有 `sync_items` 记录都有本地 inode（文件夹、已删除墓碑）。
- 生命周期独立：inode 映射每次扫描重建/更新；`sync_items` 是长期基线。
- 可安全清空重建：整个表丢失只让本轮重命名退化为删+增，下一轮自动恢复，不影响同步基线（见 `11` §6.3）。

#### 6.5.2 `free_up_staging` 表（释放空间事务）

替代原 `com.hwcloud.freeUpRelativePath` xattr，记录释放空间的中断恢复信息：

| 字段 | 类型 | 说明 |
|---|---|---|
| `staging_name` | TEXT NOT NULL (PK) | 暂存文件名（如 `.hwcloud_freeup-xxxx`） |
| `relative_path` | TEXT NOT NULL | 原始相对路径 |
| `file_id` | TEXT NOT NULL | 云端文件 ID |
| `source_mtime` | INTEGER | 原文件 mtime（回滚恢复用） |
| `source_size` | INTEGER | 原文件大小（回滚恢复用） |
| `created_at` | INTEGER NOT NULL | 创建时间（毫秒 epoch） |

> 流程改动：释放空间不再写暂存文件 xattr，而是在**同一事务内**向 `free_up_staging` 插一条记录；`recover_interrupted_free_up` 改读该表而非扫描暂存文件 xattr。暂存文件与恢复记录同事务，消除"文件已暂存但 xattr 未写"的窗口。

#### 6.5.3 identity 模块（身份映射的统一入口）

新增独立模块封装 inode 映射读写，所有身份查询都走它（对应原 `src/sync/identity.rs`，Kotlin 建议同名 object/repository）：

| 方法 | 行为 |
|---|---|
| `lookupByInode(conn, inode)` | 查询某 inode 对应的云端身份（fileId + 上次路径）。扫描时用于识别重命名——只读 DB，不碰文件 xattr，不涉及任何"补写自愈"。 |
| `upsertMapping(conn, inode, path, fileId)` | 下载/释放空间完成后**主动更新**映射（程序自己操作文件时的确定性记账，非猜测）。 |
| `purgeMissing(conn, seenInodes)` | 扫描结束后，根据本轮见到的 inode 集合清理陈旧记录。 |

#### 6.5.4 唯一新增兜底：下载覆盖

inode 方案诚实承认一处兜底。下载覆盖会重建文件（换 inode），需下载器主动记账：

```
下载流程：占位符(0字节, inode=100) → 删除 → 写入真实内容(新 inode=200)
                                                        ↑
                                              inode 变了，DB 还记着 100
```

下载器写完文件后主动 `upsertMapping` 新 inode。与原方案"xattr 补写失败静默丢失身份"不同，DB 事务内更新要么成功要么回滚，是确定性记账（见 `11` §5）。

## 7. 云端树缓存结构（CloudTreeCache）

文件 `cloudtree_<escaped>.json`（escape 规则见 §8）：

```rust
struct CloudTreeCache {
    root_folder_id: String,                       // 根目录 fileId
    tree: HashMap<String, DriveFile>,             // rel_path → 云端文件完整对象
    path_to_id: HashMap<String, String>,          // rel_path → fileId（双向索引）
    cursor: String,                               // changes 增量游标
    complete: bool,                               // 是否完整 BFS（哨兵）
}
```

### 7.1 validate_trusted 完整校验

载入后必须通过 `validate_trusted` 才能被 planner 信任，全部条件**与**关系：

1. `complete == true`（BFS 已完整覆盖）
2. `cursor` 非空
3. 每个路径键**非空**
4. 每个 `DriveFile.id`（fileId）**非空**
5. 所有 fileId **唯一**（去重后数量一致，无同一文件出现在两个路径）
6. `path_to_id[path] == tree[path].id` **双向一致**（双向索引同步）
7. **空路径 → root_folder_id**（根目录特殊映射）

任一条件失败 → **fail closed**（视为不可信，回退全量 BFS 重建）。

### 7.2 原子写

候选文件 `.json.tmp`：
1. 写 `.json.tmp`
2. `fsync(.json.tmp)`
3. `rename(.json.tmp → .json)`
4. `fsync(父目录)`（确保 rename 持久化）

崩溃恢复时若看到 `.json.tmp` 残留则丢弃。

### 7.3 备份与哨兵

- `.json.bak` 保留上一份完整 cache，主文件损坏时回退。
- `complete = false` 是哨兵：planner **拒绝使用**，触发全量重建。
- **严格完整空盘是合法 checkpoint**（`complete=true` 且 `tree` 为空），不会触发误重建。

## 11. paths.rs 安全函数（防穿越）

所有接受用户/云端路径输入的接口必须经过这三道关卡：

| 函数 | 拒绝条件 |
|---|---|
| `validate_path_segment(name)` | 空 / `.` / `..` / 含 `/` / 含 `\` / 含 `\0` |
| `validate_relative_path(rel)` | 绝对路径（以 `/` 开头）/ 含 `..` 上跳 / 含 `\` / 含 `//`（连续斜杠）/ 非 UTF-8 |

## 12. PENDING_FILE_ID_PREFIX 机制

新增文件上传**首次入队时尚无真实云端 fileId**——但 sync_items 主键要求 `file_id` 非空。用占位前缀解决：

```rust
const PENDING_FILE_ID_PREFIX: &str = "pending:";

// 入队时 file_id 暂填
format!("pending:{}", rel_path)   // 例: "pending:docs/readme.md"
```

### 流转

1. **入队 / 首次失败**：sync_items 行的 `file_id = "pending:<rel_path>"`，`status = SYNCING / FAILED`。
2. **上传成功**：拿到真实 fileId 后，**先清理可能的 pending 孤儿行**：

   ```sql
   DELETE FROM sync_items WHERE local_path = ? AND file_id = 'pending:' || ?;
   ```

   再 `upsert` 真实 fileId 行。

> 这样保证：同一本地路径不会同时存在 pending 行和真实行两条记录，避免 `find_by_file_id` 的"拒绝歧义基线"报错（见 §2.4）。

| 模块 | 函数数 | 职责 |
|---|---|---|
| `repository/sync_items.rs` | 6 | sync_items 表 CRUD（最后成功基线读写）：`find_by_file_id` / `load_all` / `upsert` / `delete_by_local_path` / `reset_stale_statuses` 等 |
| `repository/transfer_queue.rs` | — | transfer_queue 表 CRUD + **CAS 乐观并发控制**：`transition_transfer` / `transition_transfer_clearing_upload_session` / `patch_transfer_in_state` / `update_running_transfer` / `prune_transfer_history` |

### Kotlin 重构建议

- 用 **Room DAO** 或 **SQLDelight**，迁移文件逐版本保留（v2 / v3 / v4 / v5 / v6 各一份；v6 为 inode 方案的纯增量建表，见 §5.5）。
- CAS revision **不要换成 `synchronized` 锁**：跨进程/跨线程的迟到回调场景下，乐观锁 + WHERE 条件是正确性保证。
- ColumnPatch 三态语义在 Kotlin 中可用 sealed class 表达。
- 状态迁移表 `can_transition` 建议用 `EnumSet<TransferState>` 的 `Map<State, EnumSet<State>>` 静态初始化，避免遗漏。
