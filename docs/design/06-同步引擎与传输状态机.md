## 2. Planner — 3-way diff 决策（24 种情况）

文件：`src/sync/planner.rs`（439 行）

### 2.1 输入数据结构

> **inode 方案**：`LocalFileEntry` 新增 `inode: Long` 字段（来自 `stat().st_ino`）；文件身份查询走 `identity.lookupByInode`（DB inode 映射），不再读取 xattr。fileId 来源改为 DB inode 映射，详见 `10-基于inode的文件身份识别方案.md`。

```kotlin
data class SyncSnapshot(
    val local: Map<String, LocalFileEntry>,       // 本地扫描（每个 entry 含 inode）
    val cloud: Map<String, DriveFile>,            // 可信云树快照
    val db: Map<String, DbSnapshotEntry>,         // sync_items 基线
    val cloud_tree_trusted: Boolean,              // false → 抑制任何删除
    val is_startup_resume: Boolean,               // 启动恢复期标志
)

data class LocalFileEntry(
    // ... 现有字段（size/mtime/is_placeholder 等）...
    val inode: Long,                              // inode 方案新增：来自 stat().st_ino，身份识别依据
)

data class DbSnapshotEntry(
    val file_id: String,                          // 真实 fileId 或 "pending:<rel_path>"
    val local_mtime: Long?,
    val local_size: Long?,
    val cloud_edited_time: Long?,
    val status: Int,                              // SYNCED/CLOUD_ONLY/FAILED/DELETED/...
    val is_folder: Boolean,
)
```

### 2.2 SyncActionType 枚举（10 种）

| 类型 | 含义 |
|---|---|
| `Upload` | 上传本地文件到云端 |
| `CreatePlaceholder` | 创建本地占位符（云端有、本地无） |
| `Download` | 下载云端文件到本地 |

### 2.3 变更判定（精确算法）

```kotlin
// 本地是否变更（mtime 或 size 与 DB 不同）
fun is_local_changed(local: LocalFileEntry, db: DbSnapshotEntry): Boolean {
    if (db.local_mtime == null) return true       // 首次记录视为已改
    if (local.mtime != db.local_mtime) return true
    db.local_size?.let { if (local.size != it) return true }  // v3：mtime 精度不足兜底
    return false
}

// 云端是否变更（仅比较 editedTime，云端时间为权威基准）
fun is_cloud_changed(cloud: DriveFile, db: DbSnapshotEntry): Boolean {
    val cloudMs = cloud.edited_time?.timestampMillis() ?: return false  // 无时间→未变（保护）
    if (db.cloud_edited_time == null) return true                        // DB 无基线→已变
    return cloudMs != db.cloud_edited_time                               // 仅比 editedTime，不比 size
}
```

### 2.5 不可信 cloud_tree 守卫

`plan()` 顶层过滤（planner.rs:62-74）：
```
!cloud_tree_trusted && (DeleteFromLocal | DeleteFromCloud) → 丢弃动作并 warn
```
**注意：只抑制删除**，上传/下载/占位仍可执行。

### 2.6 pending 占位项收敛

- `PENDING_FILE_ID_PREFIX = "pending:"`
- 第 6 项：三方都在 + DB 是 pending → 发 `Skip + cloud_file`，让 engine 在 `apply_results` 里 upsert 真实 fileId + 清理 pending 孤儿行
- 第 13 项：本地有+云端无+pending+非 FAILED → 重新 Upload
- **第 12 项：FAILED 状态 pending 不自动重试**（避免空间不足导致无限重试循环），留给用户在传输面板手动点"重试"

> **inode 方案**：占位项收敛涉及的 fileId 来源说明——决策表中 `file_id` 的判定（pending/真实）逻辑不变，但"真实 fileId"来自 DB inode 映射（`identity.lookupByInode`）而非读取文件 xattr。24 种决策本身的触发条件与动作语义完全不变。

## 3. Executor — 并发执行

文件：`executor.rs`（132）+ `actions.rs`（720）+ `local_delete.rs`（232）+ `transfer_operations.rs`（717）

### 3.1 并发执行（execute_all）

- `Semaphore::new(concurrency.max(1))`（默认 6，可配 1-20）
- `stream::iter(...).map(...).buffer_unordered(concurrency).collect()` 并发跑
- 每个动作拿到 slot 后必须通过 `begin_action_activity` 门（engine shutdown 后返回 deferred=true，不记 FAILED）
- `catch_unwind` 捕获 panic，按原 action_id 记失败
- 结果按 action_id 排序还原顺序
- 执行前先 `prune_transfer_history(100)`（保留最近 100 条终态）

### 3.2 目录优先执行 + path_to_id 回填（两阶段）

**阶段 1**：收集所有 `CreateFolder && cloud_file.is_none()`（本地→云端新建目录），按路径深度（`/` 计数）升序**顺序执行**。每个成功后立即 `apply_results` 结算 DB + 回填 `path_to_id`，再广播 `content_changed=true`。

**阶段 2**：用最新 `path_to_id` 调 `fill_parent_file_ids` 回填其余动作的 `parent_file_id`（取 `relative_path.rfind('/')` 前缀查 p2i map），然后并发执行剩余动作。

### 3.3 稳定性检查接入

仅在 Upload/Create/Update 前调用。重试窗口 `[0, 2, 3, 5]` 秒（共 4 次）：
- `StabilityChecker.check()` 返回 `Stable` → 通过
- `Editing` → `restart_required("用户正在编辑")`
- `Unstable` → 继续下一个 delay
- 全部失败 → `restart_required("文件尚不稳定")`

### 3.4 冲突解决（do_conflict）

调用 `ConflictResolver.resolve(local_path, cloud_file, local_mtime)`：
- **Cloud 胜**：`rename(local → copy_path)` → 若失败直接返回失败（保本地修改）；成功后 download 云端新版到 local_path + mark_downloaded
- **Local 胜**：先 download 云端旧版到 copy_path 做副本，再 upload_update 覆盖云端

> **inode 方案**：原 `do_conflict` 中的 `clear_placeholder_xattr` 相关调用已删除。副本通过 `rename`/`download` 天然产生**新 inode**，DB 自动当作独立文件处理，无需再"撕掉副本上的 fileId xattr"防止冒充原件。详见 `10-基于inode的文件身份识别方案.md` §4.6。

### 3.5 上传成功后回写

执行阶段：上传返回 `edited_time.is_none()` → `files_api.get(id)` 补全元数据，否则 disposition=VerifyingRemote。

> **inode 方案**：原 `set_upload_file_id_if_current`（再次 `verify_source_snapshot` 后补写 xattr fileId，源已变则跳过）**已删除**。改为 `identity.upsertMapping`：在 DB 事务内写 inode 映射（inode + relative_path + fileId）。这是程序自己操作文件时的**确定性记账**（非"补写自愈"），随基线结算同事务原子完成——要么成功要么回滚，不再有"xattr 补写失败导致身份静默丢失"的故障模式。详见 `10-基于inode的文件身份识别方案.md` §4.7。

结算阶段（`settle_success`，原子事务）：
- 上传：用 `running.source_mtime/source_size`（任务持久化时的快照，**不是当前 stat**）写 sync_items
- 下载：用当前 stat
- `DELETE FROM sync_items WHERE local_path=? AND file_id='pending:<path>'` 清 pending 孤儿
- Update 操作额外 `DELETE FROM sync_items WHERE file_id=? AND local_path<>?` 清旧路径
- upsert SYNCED + 清 error

### 3.6 进度推送（progress.rs）

`TaskProgressReporter` 持 `running_revision` 做门禁：
- `update_transferred(transferred)` / `update_download_progress(transferred)` / `update_resume(server_id, upload_id, offset, session_url)`
- **节流 500ms**（`PROGRESS_THROTTLE_MS`）
- 越界检查（transferred<0 || >total_size → Err）
- 每次持久化后 `publish_state_best_effort`

### 4.3 CAS revision 乐观锁

```sql
UPDATE transfer_queue SET
  state=?1,
  error_kind=CASE ?2 WHEN 0 THEN error_kind WHEN 1 THEN ?3 ELSE NULL END,
  ...（error_message/next_retry_at/finished_at/remote_result_file_id/session_url 类似）

  resume_offset=CASE WHEN ?15 IS NULL THEN resume_offset ELSE ?15 END,
```

### 4.5 数据安全关键规则（真实实现）

#### 上传恢复（recovery.rs + transfer_operations.rs）
- 进程重启后遇到 Running 上传：**绝不推算 offset = resume_offset + chunkLen**
- 改为迁移到 `VerifyingRemote`（"进程中断时远端写入结果不确定，等待核验"）
- 由 `resume_verifying` → `verify_remote` 用 GET/list_all 核验真实结果
- 带 session_url 的续传：`on_resume` 回调把**服务端返回的 rangeList 确认的 offset** 回写 DB，向前推进只认服务端确认的范围

#### 下载恢复（recovery.rs）
- `.tmp` + 版本 sidecar（`.download-meta.tmp`）
- `durable_offset = metadata(&.tmp).len().min(total_size)`（**只认 .tmp 实际文件长度，不推算**）
- 迁移 RestartRequired(SessionExpired) → Pending，带 `transferred=durable_offset, resume_offset=durable_offset`

#### 写操作结算（禁止盲目重放）
1. **成功核验（validate_success_outcome）**：
   - 上传：必须有 `cloud_file.id` 非空 + name 匹配 + edited_time 有值 + size 匹配 + Update 时 file_id 匹配，否则 remote_ambiguous
   - 下载：`metadata.len() == total_size` + expected_cloud_edited_time 有值
2. **响应丢失 GET 核验（verify_remote）**：
   - Create：优先用 remote_result_file_id GET；404→Ambiguous（"禁止重复创建"）；否则 list_all 按 name+size+parent+created_time 窗口（`created_at ± 120s..+30天`）匹配 + content_hash 比对
   - Update：GET file_id，比对 edited_time != expected → NotCommitted；身份一致 → Committed
3. **RestartRequired 含 remote_result_id 自动提升**（admission.rs）：任何带 persisted remote result 的 RestartRequired 在入队时强制转 VerifyingRemote，禁止重放

#### 启动恢复固定顺序（8 步）
```
1. reset_stale_statuses（SYNCING→失败，清孤儿）
2. load_or_refresh_cloud_tree（先建可信 checkpoint）
3. promote_ambiguous_restarts（RestartRequired+remote_id → VerifyingRemote）
4. recover_verified_remote_path_changes（路径恢复，收敛远端改名）
5. purge_deleted_tombstones_if_trusted
6. STARTUP：recover_interrupted_transfers → commit_recovery_checkpoint
7. ONLINE_RECOVERY：resume_verifying → resume_waiting → resume_due_backoff
   （严格此顺序：先核验、再网络、最后退避）
8. run_sync_cycle_inner（本地扫描 + planner + executor）
```

`recover_startup` 内部：按 `created_at` 降序分组同 path，Running+写操作 → 全部 VerifyingRemote；选最新一个执行，其余 suppress。

### 5.2 分类决策树（classify_transfer_error）

```
QuotaExceeded → permanent(Quota)
Token NotLoggedIn/RefreshFailed / Auth → permanent(Auth)
upload_session_expired error_code → SessionExpired + VerifyRemote（不消耗预算）
DriveApi 带 transport_kind → classify_transport:
  Connect → Network + WaitForNetwork（不消耗预算）
  写操作 + request_may_have_reached_server → verify_remote()（RemoteAmbiguous）
  Network/Timeout/ResponseBody → WaitForNetwork（不消耗）
  Decode → 预算耗尽 permanent(Server)，否则 Backoff
  Request/Other → permanent(Unknown)
DriveApi HTTP 状态:
  401（未重放）→ RefreshAuth；401（已重放）→ permanent(Auth)
  429 → Backoff（Retry-After 或指数退避）；预算耗尽 → permanent(RateLimit)
  500/502/503/504 → Backoff；预算耗尽 + 写操作 + 可能到达 → VerifyRemote，否则 permanent(Server)
  400/409/422 → permanent(Validation)
  403 → permanent(Permission)
  其他 → permanent(Unknown)
Config/Generic → permanent(Unknown)
```

### 5.3 退避算法（exponential_backoff_at）

```kotlin
val exponent = attemptCount.coerceAtMost(63)
val seconds = (1uL shl exponent).coerceAtMost(300)     // 上限 300s = 5min
val delayMs = (seconds * 1000 + jitterMs).coerceAtMost(300_000)
val nextRetryAt = nowMs + delayMs
```

- `MAX_AUTOMATIC_ATTEMPTS = 5`（settlement.rs:14）
- 退避序列：1s, 2s, 4s, 8s, 16s（带上限 300s）
- `budget_exhausted`：`attempt_count >= max_attempts`

### 5.4 preflight 预检

**静态校验（validate_static）**：
- 路径合法性、挂载根存在、绝对路径 == mount_root+rel
- total_size/resume_offset >= 0，resume_offset <= total_size
- Create 不能带 fileId；Update 必须真实 fileId（非 pending:）
- **非零 resume_offset 必须有 session_url**（拒绝作为全新请求重放）
- 上传源 mtime+size+total_size 必须匹配任务快照
- 下载目标不能是目录、不能有非占位内容

**后端 preflight**：稳定性检查 + Update 时 `total_size > 20MB` → restart_required

## 14. inode 方案下的对账与路径恢复

> 本节描述 inode 方案（`10-基于inode的文件身份识别方案.md`）下，`reconciliation` 与 `path_recovery` 子模块的现行架构。原 fileId xattr 兜底逻辑被结构性移除。

### 14.1 改名检测：detect_renames → detect_moves

⚠️ **inode 方案**：`detect_renames`（~180 行，含复制消歧、延迟替换源跟踪、"旧路径被其他文件占用"分支）**重写为 `detect_moves`**（基于 inode 映射配对，核心逻辑约 40 行）。

核心逻辑：

```text
对每个"未知 inode + 新路径"的文件：
  old = identity.lookupByInode(conn, inode)   # 查 DB：这个 inode 上次在哪？
  if old 存在 AND old.relativePath != 新路径:
      # 同一个 inode 在新路径出现 = 移动！
      action = MoveInCloud { fileId = old.fileId, relativePath = 新路径, ... }
      identity.upsertMapping(conn, inode, 新路径, old.fileId)   # 更新映射
```

**与原方案的关键差异**：完全不需要"老路径还在不在"的消歧。因为 `cp` 复制产生新 inode，根本匹配不上老 inode——副本天然被当新文件处理，无需任何消歧分支。

二级兜底（可选）：检测到"老 inode 消失 + 新文件出现"时，若新文件 `size == 老记录 size AND mtime == 老记录 mtime`，视为疑似 inode 漂移（断电/还原所致），保守标记下一轮再确认，不立即判删除。详见 `10-基于inode的文件身份识别方案.md` §4.3 与 §6.2。

### 14.2 reconcile_db_records：删除三层 fileId 关卡

原"给无基线的本地文件建基线"需过三道 fileId 关卡（对抗"同一 fileId 多处出现"）。inode 方案下**三层关卡全部删除**：

| 原关卡 | inode 方案 |
|---|---|
| 必须有 xattr fileId | ❌ 删除（查 inode 映射即可） |
| 同路径云端 id 必须等于 xattr fileId | ❌ 删除（inode 自然配对） |
| 全表扫描确认无其他路径持同一 fileId | ❌ 删除（inode 主键唯一，不可能重复） |

简化后规则：**本地文件无基线但 inode 映射里有 fileId → 直接用该 fileId 建基线**。详见 `10-基于inode的文件身份识别方案.md` §4.4。

### 14.3 path_recovery recover_one：简化（8 检查删 6 个）

`recover_one` 现含 8+ 个检查，其中 6 个与 fileId 歧义相关。inode 方案下：

| 原检查 | inode 方案 |
|---|---|
| 同一 fileId 不能有多条基线 | ❌ 删除 |
| 读老路径 xattr 验证所有权 | 🔄 改为读 inode 映射 |
| 读新路径 xattr 验证所有权 | 🔄 改为读 inode 映射 |
| `read_file_id` 非 UTF-8 致命错误 | ❌ 删除（不读 xattr 了） |

函数预计从 ~160 行缩减到 ~50 行。详见 `10-基于inode的文件身份识别方案.md` §4.5。

### 14.4 释放空间流程：xattr → DB 事务

原 13 步 TOCTOU 流程中，暂存文件恢复路径依赖 `XATTR_FREE_UP_RELATIVE_PATH` xattr 标记。inode 方案下**不再写 xattr**，改为在 DB 事务内向 `free_up_staging` 表插入一条记录（`staging_name` / `relative_path` / `file_id` / `source_mtime` / `source_size`）。

- `recover_interrupted_free_up` 改为读 `free_up_staging` 表，不再扫描暂存文件 xattr。
- 回滚恢复逻辑不变，数据来源从 xattr 改为 DB。
- **好处**：暂存文件和恢复记录在同一 DB 事务内，消除"文件已暂存但 xattr 未写"的窗口。详见 `10-基于inode的文件身份识别方案.md` §4.8。
