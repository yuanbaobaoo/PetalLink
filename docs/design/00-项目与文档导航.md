# PetalLink Flutter：项目与文档导航

> **来源**：petal-link-cmp `docs/plan/00`（2026-07 纳入 Flutter 项目并按本项目重写）。

## 1. 项目目标

PetalLink 是一个面向 macOS 的华为云空间桌面客户端，通过华为 Drive REST API 直连云端，把云端文件映射到本地目录并执行双向同步。

本仓库是原 Tauri 2.x 项目的 **Flutter 移植版**，保留原项目已经验证过的协议兼容性、数据安全边界和 macOS 原生体验。

- 当前仓库：`/Users/Shared/codes/personal/petal-link-flutter`
- 原始仓库（业务逻辑基线）：`/Users/Shared/codes/personal/petal-link`（Rust + Tauri 2.x + Vue 3 + Pinia + SQLite）
- 参考仓库（inode 方案来源）：`/Users/Shared/codes/personal/petal-link-cmp`（Kotlin/CMP 移植，未完成）
- 目标平台：macOS 优先
- 目标技术栈：Flutter + Dart + GetX + Dio + sqflite + MethodChannel（macOS 原生桥接）

**移植原则**（2026-07-21 确立）：

1. **业务逻辑以 Tauri/Rust 原版为准**——协议合同、同步语义、错误恢复、状态机全部对齐 Rust 源码。
2. **唯一例外：文件身份识别采用 CMP 设计的 inode 方案**（见 `10-基于inode的文件身份识别方案.md`），替代 Tauri 的 fileId xattr 方案。xattr 仅保留 `com.hwcloud.state` 与 `com.apple.FinderInfo` 两个键。
3. CMP 与 Tauri 冲突的其他实现细节（错误模型分类、sync_items 主键形态等）**不随 CMP**，维持 Tauri 语义。

"完整移植"是指用户可见功能、后台同步语义、错误恢复、安全合同和发布能力达到原项目等价，而不是仅完成类名或模块的一一翻译。

## 5. 不可弱化的移植约束

1. 华为 API 已验证怪癖必须保留，包括 scope `/` 不编码、授权码 `+` 编码、中文 JSON `\uXXXX`、`multipart/related`、308 `rangeList`、`nextCursor` 与 `newStartCursor` 分离。
2. 写操作只有在核验 `HTTP 200 + File` 或远端 GET 收敛后才能结算；不确定写禁止盲目重放。
3. 九态传输状态机必须执行合法迁移校验和数据库 CAS revision。
4. 上传恢复只认服务端确认 offset；下载恢复只认 `.tmp` 实际长度与版本 sidecar。
5. **文件身份由 `local_inode_map` 维护（inode 方案）；xattr 只保留 `com.hwcloud.state` 与 `com.apple.FinderInfo`**。
6. 无 state xattr 的文件属于用户，绝不转换；用户 0 字节文件绝不因 size 为 0 被删除。
7. 本地监听必须使用 FSEvents 语义、3 秒 debounce、2 秒 warmup、纯事件驱动和 generation 隔离。
8. `.hwcloud_` 前缀在扫描、监听、DB、planner 和 UI 中统一隔离。
9. 网络恢复顺序固定为 cloud catch-up → VerifyingRemote → planner。
10. 关闭窗口和普通退出应进入后台；只有托盘"退出 PetalLink"执行真实退出。

## 6. 工作方式

每个实施阶段必须遵循以下闭环：

1. 先从 Tauri 原仓库源码和 `docs/design/` 提取合同。
2. 先补失败测试，再实现真实路径。
3. 单元测试、合同测试和集成测试全部通过（`flutter test`）。
4. 从应用入口验证功能已经装配，而不是只有孤立类存在。
5. 涉及 inode 落地的阶段，按 `docs/review/2026-07-21-inode-scheme-evaluation.md` §六的五阶段计划推进并更新完成状态。
