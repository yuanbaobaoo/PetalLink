# 13 · 发布与兼容验收

> **来源**：petal-link-cmp `docs/`（2026-07 纳入 Flutter 项目）。术语映射：CMP ViewModel ≈ Flutter GetX Controller；Room ≈ sqflite；StateFlow ≈ Rx/Stream。文件身份识别以 **inode 方案**为准（见 `10-基于inode的文件身份识别方案.md`），文中如涉及旧 xattr fileId 描述均为历史背景。


## 1. 发布输入

- 唯一版本：根目录 `version.properties` 的 `petalLinkVersion`
- 构建档案（dev/release）：由 `-Prelease` 布尔开关决定（不带 / false / 非 true → dev）。它是 bundle id、运行时数据目录与 LaunchAgent label 的**唯一真相源**：
  - **默认 dev**：`run`/`jvmTest`/`packageDmg` 均落到 dev 数据目录，绝不污染正式用户数据
  - `-Prelease=true` → release，bundle id `io.github.yuanbaobaoo.PetalLink`（沿用原 Tauri，老用户数据目录不变）
  - 不带 → dev，bundle id `io.github.yuanbaobaoo.PetalLink-dev`（数据目录、单实例锁、开机自启与 release 完全隔离）
  - 打包期写进 `.app` 的 `CFBundleIdentifier`，同名常量 `BuildInfo.BUNDLE_ID` 编译进 JVM；运行时 `AppPaths` 默认读 `BuildInfo.BUNDLE_ID` 派生数据目录，两个开关合一，消除「打了 dev 包却误读 prod 数据」类 bug
- **正式发布必须使用 `./kotlin do releaseDmg`**；本地预览使用 `./kotlin do packageDmg`（默认 dev）
- 最低系统：macOS 12.0
- 架构：当前发布为 arm64
- 更新 endpoint：`BuildInfo.UPDATE_ENDPOINT`
- 正式构建必须注入 `PETALLINK_UPDATE_TEAM_ID`，否则客户端 fail closed 拒绝安装更新

## 3. 自动构建

客户端更新检查保留原版节奏：启动 3 秒后静默检查、每小时检查、窗口聚焦按 10 分钟节流，设置页手动检查不受节流；发现新版本后在主界面和设置页提供入口。

CI 执行以下步骤：

1. `jvmTest`：单元、HTTP 合同、临时目录集成和 macOS 平台测试；当前 334 项通过。
2. `packageDmg`：实际生成无签名 DMG。
3. Release 额外执行 Developer ID 签名、notarytool 和 staple。
4. `.app.zip` 生成 SHA-256 manifest；客户端安装前仍由更新器核验 SHA-256、codesign、spctl 与固定 Team ID。

CI 在 Flutter 打包命令（`flutter build macos`）成功后直接上传产物，不执行打包产物静态校验或启动冒烟。

## 4. 原 Tauri 数据兼容

Flutter 版继续使用原 bundle id，因此直接读取同一目录：

`~/Library/Application Support/io.github.yuanbaobaoo.PetalLink`

- `token.bin`：`PTL1 + nonce + ChaCha20-Poly1305` 完全沿用，明文 length-prefix 布局一致。
- `config.json`：字段与枚举字符串保持 camelCase 兼容（Flutter 侧持久化为 SQLite `config` 键值表，导入/导出格式与原 JSON 一致；从 Tauri 版升级时配置不自动迁移）。
- `petal_link.db`：sqflite 新库（schemaVersion=6，含 inode 两张表）；首次启动重新建立同步基线，不迁移旧库。
- 云树/checkpoint：完整性不足或 cursor 失效时 fail closed 并安全重建，不要求删除用户文件。

## 5. 原命令替代关系

- `drive_list_all`：`driveListAll` 保留；UI 浏览使用 cursor 分页避免大目录一次性载入。
- `drive_download_file`：UI 统一走 `syncDownloadOnDemand`，两者最终进入同一持久 TaskRunner。
- `sync_free_up_space`：多选和单选统一组装为 batch，复用同一释放空间安全协议（`service/mount/free_up.dart`）。
- `transfer_clear_completed/failed/finished`：底层保留三种精确语义，UI 用 finished 作为默认入口。
- `sync_folder_recursive`：保留“立即返回 0 + 后台执行”语义；在统一 owner 内 BFS 目标云端子树，下载真实内容、上传本地独有文件，完成后再全量核对 checkpoint。
- Tauri updater：替换为签名 `.app.zip` staging/helper；替换或启动失败自动恢复 `.backup`。

## 6. 发布后人工矩阵

以下项目依赖真实华为账号、Apple 凭据或系统 UI，必须在 Release Candidate DMG 上执行并记录版本/机器/结果：

- 新用户：安装 → 登录 → 选择空目录 → 首次索引 → 双向同步。
- 老用户：覆盖安装后恢复 token/config，sqflite 新库首次建立新基线；不迁移旧库。
- 网络：离线启动、上传/下载中断、429/5xx、401 refresh、恢复网络。
- 文件：0B、>20MiB、中文、emoji、rename/move/copy、持续编辑和占位按需下载。
- 云端：rename/move/delete、cursor 400/410、空中间页、完整空盘。
- 生命周期：Close、Cmd+Q、Dock Quit、系统关机/登出、LaunchAgent、第二实例。
- 发布：DMG Gatekeeper、签名、notarization/staple、更新成功、替换失败回滚。

人工矩阵未签字前，只能称为“构建通过”，不能称为“生产验收通过”。
