# 05 · 业务服务与事件接口规范

> **来源**：petal-link-cmp `docs/`（2026-07 纳入 Flutter 项目）。术语映射：CMP ViewModel ≈ Flutter GetX Controller；Room ≈ sqflite；StateFlow ≈ Rx/Stream。文件身份识别以 **inode 方案**为准（见 `10-基于inode的文件身份识别方案.md`），文中如涉及旧 xattr fileId 描述均为历史背景。


> 原项目通过 Tauri `invoke()` 暴露 **49 个命令 + 4 个事件广播**，由 Rust 后端与 Vue3/TS 前端跨进程桥接。
> CMP 重构后，这些「命令」将退化为 **ViewModel → 业务层的直接方法调用**（同进程，无需 JSON 序列化），「事件」退化为 **`StateFlow` / `SharedFlow` 状态流**。下表给出完整的签名接口规范，供重构时逐一对照实现。
>
> - 协议字段、SQL 列名、HTTP method、`token`、`cursor`、`checkpoint` 等专有技术名词保留官方英文与大小写。
> - 参数与返回值沿用前端 `camelCase` 调用风格；DB 列名以 `snake_case` 标注。
> - 命令清单的权威来源是 `src/lib.rs` 的 `tauri::generate_handler!`，下表按领域分组列举，合计 49 项。

## 二、DELETE_TRACE_ERROR_PREFIX（跨语言合同，重点）

软删除的「云端 PATCH 成功」与「本地传输记录留痕」是两个独立写操作，存在「文件已删但记录未写入」的边界态。为此定义跨语言共享错误前缀：

```
DELETE_TRACE_ERROR_PREFIX = "TRACE_FAILED:"   // commands/drive.rs
```

`drive_delete_file` 的处理顺序：

1. 调用华为 `PATCH /drive/v1/files/{id} {"recycled":true}`，严格核验 `HTTP 200 + 同 fileId + recycled=true`，否则按真失败处理。
2. **删除成功后** 调用 `record_completed_delete`，写一条 `Completed` 状态的删除留痕到 `transfer_queue`，保证主页传输面板可见。
3. **若留痕写入失败**，命令返回字符串错误：

   ```
   {DELETE_TRACE_ERROR_PREFIX}文件已删除，但传输记录写入失败：{error}
   ```

前端 `app/api/drive.ts` 据此前缀区分两种语义完全不同的失败：

| 错误形态 | 含义 | 前端响应 |
|---|---|---|
| 不含 `TRACE_FAILED:` 前缀 | **文件未删**（PATCH 真失败 / 未核验通过） | 弹错误 toast，文件仍存在 |
| 含 `TRACE_FAILED:` 前缀 | **文件已删但记录未写入** | 不重复发起删除；提示用户记录缺失，可手动刷新列表 |

**跨语言合同测试**：`app/tests/drive.contract.test.ts` 校验前端常量与 Rust 源码字面量完全一致，**任何一侧改动必须同步另一侧**，否则测试失败阻断 CI。

## 三、事件广播（4 个 + 1 个辅助）

所有事件由 Rust 后端通过 `app.emit(name, payload)` 推送，前端在 `app/main.ts` 注册全局监听。

### 3.1 主事件清单

| 事件 | payload | emit 入口 | emit 时机 |
|---|---|---|---|
| `sync_state` | `SyncGlobalState` | `emit_sync_state` | 状态桥接任务收到 engine 广播时；`transfer_clear_*` 后；`sync_state` 命令同步返回前 |
| `folder_content_changed` | `()` | `emit_folder_content_changed` | 状态桥接发现 `state.content_changed==true` 时；`drive_delete_file` 删除成功后 |
| `transfer_update` | `()` | `transfer_update_tx` 广播 → 监听器 emit | 任务入队 / 进度推进 / 结算 / 引擎状态推送；`drive_delete_file` 留痕后 |
| `upload_failed` | `{ name: string, error: string }` | `executor/transfer_operations.rs` | 上传执行捕获错误时（executor 持 `app_handle` 直接 emit，前端弹 toast） |

### 3.2 辅助事件

- `folder_sync_progress`：payload `{ done: number, total: number }`，由 `sync_folder_recursive` 的后台任务在**每项子任务完成时** emit，用于目录递归同步进度条。

### 3.3 状态桥接任务实现（`ensure_engine_started` 内 spawn）

桥接任务把 SyncEngine 的进程内 broadcast channel 转发为前端事件，是 CMP 重构的核心替换点：

```
ensure_engine_started()
  → state_receiver = engine.subscribe()       // ① 先订阅 state_receiver
  → engine.start()                            // ② 再 start engine
                                              //    （顺序关键：保证启动期状态也转发）
  → spawn {
      loop {
        match rx.recv() {
          Ok(state) => {
            emit_sync_state(state)            // ③ 推送 SyncGlobalState 到前端
            tray::refresh_menu(state)         // ④ 同步刷新托盘菜单
            if state.content_changed {
              emit_folder_content_changed(()) // ⑤ 触发前端目录重拉
            }
          }
          Err(Lagged(n)) => warn!(...),       // ⑥ 消费慢导致的丢帧：告警继续
          Err(Closed)   => break,             // ⑦ channel 关闭：退出任务
        }
      }
    }
```

> 「先订阅再 start」是不可妥协的顺序约束——若颠倒，启动期 BFS / 首次 cycle 产生的状态广播会因为没人订阅而丢失，前端永远拿不到 `indexing-startup` 阶段的状态。

## 四、SyncGlobalState 完整结构（权威快照）

```kotlin
data class SyncGlobalState(
    val revision: Long,              // 进程内单调版本（前端拒绝旧快照）
    val total: Long,
    val completed: Long,             // = total - failed - conflict
    val uploading: Long,
    val downloading: Long,
    val waitingNetwork: Long,        // 等待网络恢复（非永久失败）
    val failed: Long,
    val transferFailed: Long,        // 传输队列永久失败历史数
    val failedItems: List<FailedItem>, // 最多 20 条
    val conflict: Long,
    val editing: Long,               // 被暂停编辑的文件数
    val isRunning: Boolean,
    val lastSyncTime: Long?,         // 毫秒 epoch
    val isIndexing: Boolean,
    val indexingScannedFolders: Long,
    val indexingDiscoveredItems: Long,
    val contentChanged: Boolean,     // 触发前端目录重拉
    val syncPhase: String?,          // 见下表
)

data class FailedItem(
    val relativePath: String,
    val errorMessage: String?,
)
```

### 4.1 `sync_phase` 取值（供状态条精确显示）

| 值 | 含义 |
|---|---|
| `indexing-startup` | 启动索引 |
| `indexing-manual` | 手动触发索引 |
| `indexing-auto-full` | 自动全量索引 |
| `querying-changes` | 查询增量变更 |
| `syncing-auto-incremental` | 自动增量同步 |
| `syncing-local` | 本地变更同步 |
| `syncing-manual` | 手动同步 |
| `syncing-retry` | 重试同步 |
| `syncing-startup` | 启动同步 |
| `null` | 空闲 |

### 4.2 前端校验规则（`isSyncGlobalState`）

前端只接受**字段完整且 revision 不旧于当前值**的快照，缺字段、默认对象或旧 revision 均拒绝：

- 所有计数器（`total` / `completed` / `uploading` / `downloading` / `waitingNetwork` / `failed` / `transferFailed` / `conflict` / `editing` / `indexingScannedFolders` / `indexingDiscoveredItems`）为非负整数
- `revision` ≠ 0
- `failedItems` 是数组，且每项有 `relativePath: string` + `errorMessage: string | null`
- `isRunning` / `isIndexing` / `contentChanged` 为 `boolean`
- `lastSyncTime` 为 `number | null`
- `syncPhase` 为 `string | undefined`

## 五、StatusAggregator 聚合 SQL（补充）

`SyncGlobalState` 的所有计数器由 `StatusAggregator.snapshot(conn, runtime)` 在**同一 publication lock 下、单次 `query_row`** 内聚合，保证快照原子性。详见 `06-同步引擎与传输状态机.md` §12。

### 5.1 计数器查询（单次 `query_row` 7 个子查询）

```sql
SELECT
  (SELECT COUNT(*) FROM sync_items),                                                                    -- total
  (SELECT COUNT(*) FROM sync_items               WHERE status = FAILED),                                -- failed
  (SELECT COUNT(*) FROM sync_items               WHERE status = CONFLICT),                              -- conflict
  (SELECT COUNT(*) FROM transfer_queue           WHERE state = Running       AND direction = UPLOAD),                                  -- uploading
  (SELECT COUNT(*) FROM transfer_queue           WHERE state = Running       AND direction IN (DOWNLOAD, DOWNLOAD_UPDATE)),            -- downloading
  (SELECT COUNT(*) FROM transfer_queue           WHERE state = WaitingForNetwork),                       -- waiting_network
  (SELECT COUNT(*) FROM transfer_queue           WHERE state = Failed)                                    -- transfer_failed
```

### 5.2 失败项明细（独立查询）

```sql
SELECT local_path, error_message
FROM sync_items
WHERE status = FAILED
ORDER BY local_path
LIMIT 20;
```

> 明细与计数器分开查询：计数走 `COUNT(*)` 可走索引，明细需要回表，**最多 20 条**避免前端列表过长。`LIMIT 20` 与 `failedItems` 字段约定一致。

### 5.3 派生字段

- `completed = total - failed - conflict`（不在 SQL 中算）
- `revision = next_revision.fetch_add(1) + 1`（`StatusAggregator` 内 `AtomicU64`，**单调递增**）

> `revision` 由聚合器在 snapshot 内分配，**与 DB 事务解耦**——即使 DB 读取失败，下一次成功读取时 revision 仍会前进，确保前端永远不会收到「revision 倒退」的快照。

## 六、RuntimeStatus（补充）

`RuntimeStatus` 是 `SyncGlobalState` 中**非持久化的运行时字段集合**，反映引擎当前生命周期门状态，而非 DB 落盘数据：

```kotlin
data class RuntimeStatus(
    val editing: Long,                 // 被暂停编辑的文件数
    val is_running: Boolean,           // 引擎是否在执行 cycle
    val last_sync_time: Long?,         // 最后一次成功同步时间（毫秒 epoch）
    val is_indexing: Boolean,          // 是否在索引中
    val indexing_scanned_folders: Long,// 索引已扫描目录数
    val indexing_discovered_items: Long,// 索引已发现条目数
    val content_changed: Boolean,      // 本轮是否产生了目录结构变化
    val sync_phase: String?,           // 当前同步阶段（见 §4.1）
)
```

### 关键不变量：生命周期门优先

即使数据库快照读取失败（DB 锁、文件损坏等），`update_runtime_and_broadcast` 仍会调用 `runtime.apply_to(state)` **强制覆盖生命周期字段**（`is_running` / `is_indexing` / `sync_phase` / `last_sync_time` 等），保证：

- UI 永远不会在「引擎实际运行中」显示「空闲」状态
- 崩溃恢复后第一次状态广播能立即反映真实运行态，不依赖 DB 重建

只有计数器（`total` / `failed` / ...）在 DB 失败时退化为上一次成功值，不会让 UI 显示假数据。

## 七、前端乱序保护（transfer store）

前端 `transfer_list_all` 是异步轮询，多个请求可能乱序返回；为避免旧响应覆盖新状态，`loadAll()` 实现两重保护（Kotlin 重构需等效实现）：

1. **`requestId` 递增（请求级保护）**：每次发起 `loadAll()` 自增 `requestId`；响应返回时若 `requestId < lastAppliedLoadRequest`，直接丢弃，不更新 store。
2. **`revision` CAS（任务级保护）**：即使两个请求乱序返回，对**同一 task** 也比较 `state_revision`，只有更新（更大）的 revision 才允许回写，避免旧 `state_revision` 把任务状态倒退回 `Running` 等。

CMP 重构建议直接复刻这套语义，对应 Kotlin 实现：

```kotlin
class TransferStore {
    private val lastAppliedLoadRequest = AtomicLong(0)
    private val taskRevisions = ConcurrentHashMap<Long, Long>()

    suspend fun loadAll() {
        val reqId = lastAppliedLoadRequest.incrementAndGet()
        val tasks = api.transferListAll()
        if (reqId < lastAppliedLoadRequest.get()) return // 丢弃乱序响应
        tasks.forEach { t ->
            val prev = taskRevisions[t.id]
            if (prev == null || t.stateRevision > prev) {
                taskRevisions[t.id] = t.stateRevision
                upsert(t)
            }
        }
    }
}
```

## 八、全局运行时单例（`commands.rs`）

所有命令的依赖以全局 `Lazy` 单例方式注入，避免在每个命令中传 `State<>` 参数。`commands.rs` 是这套依赖装配的权威入口。

### 8.1 `Lazy` 静态单例（应用生命周期内不变）

| 单例 | 类型 | 注入时机 |
|---|---|---|
| `AUTH_SERVICE` | `AuthService` | 启动期初始化，持 `token.bin` 加密读写能力 |
| `DRIVE_CLIENT` | `DriveClient` | 启动期初始化，reqwest + 401 自动刷新重放 |
| `FILES_API` | `FilesApi` | 依赖 `DRIVE_CLIENT` |
| `CHANGES_API` | `ChangesApi` | 依赖 `DRIVE_CLIENT` |
| `DOWNLOAD_API` | `DownloadApi` | 依赖 `DRIVE_CLIENT` |
| `UPLOAD_API` | `UploadApi` | 依赖 `DRIVE_CLIENT` |
| `THUMBNAIL_API` | `ThumbnailApi` | 依赖 `DRIVE_CLIENT` |
| `DB` | Room KMP + bundled SQLite | 启动期打开 `petal_link_room.db` |
| `STATUS_AGGREGATOR` | `StatusAggregator` | 持 `next_revision: AtomicU64` + `publication: Mutex` |

### 8.2 `Mutex<Option<T>>` 动态注入（随账号/目录切换）

| 字段 | 类型 | 注入入口 |
|---|---|---|
| `MOUNT_MANAGER` | `Mutex<Option<MountManager>>` | `set_mount_manager(...)`：用户首次配置挂载目录时 |
| `SYNC_ENGINE` | `Mutex<Option<SyncEngine>>` | `set_sync_engine(...)`：`ensure_engine_started` 时 |

> 这两个用 `Option` 包裹是因为它们依赖用户运行时配置（挂载路径、账号 token），无法在编译期或启动期就绪。`drop_runtime_async` 会把两者都置 `None`。

### 8.3 `EngineOwnershipProtocol`（引擎串行协调）

引擎的安装、替换、销毁涉及多命令协作（`config_save` 切目录、`auth_login` 切账号、`app_clear_cache` 全清），必须串行，否则会出现「新引擎刚装上就被旧引擎的关闭流程干掉」的竞态。

```
EngineOwnershipProtocol {
    gate: Mutex<()>,                 // 串行闸：同一时刻只有一个 owner 在操作
    replacements: AtomicUsize,       // 替换计数：旧 owner 检测到自己被替换后主动退出
}
```

- 任何想动 `SYNC_ENGINE` 的命令先 `lock(gate)`，再 `replacements.fetch_add(1)` 注册新版本号。
- 持有旧引擎的 owner 在 await 点检查「我的版本号是否已被替换」，被替换则**不执行后续关闭逻辑**（避免误关新引擎）。
- 这套协议保证切账号、切目录、清缓存等场景下引擎替换是原子的。

## 九、CMP 重构建议（补充）

Tauri 的 IPC 桥在 Compose Multiplatform (CMP) 同进程架构下不再需要——命令退化为 ViewModel 方法调用，事件退化为 Kotlin Flow。具体映射：

### 9.1 命令 → ViewModel 方法

```kotlin
class SyncViewModel(
    private val syncEngine: SyncEngine,
    private val statusAggregator: StatusAggregator,
) {
    fun syncState(): SyncGlobalState = statusAggregator.snapshot()  // 同步返回，无 invoke 开销
    suspend fun syncManualRefresh() = syncEngine.triggerManualSync()
}
```

- 不再需要 `serde` 序列化 + JSON 桥接 + 反序列化往返。
- 错误用 Kotlin `Result<T>` / sealed class 表达，等价于 Rust 的 `AppError` tagged enum。
- 参数与返回值直接用 Kotlin 类型，不需要 `as` 强转。

### 9.2 事件 → `StateFlow` / `SharedFlow`

```kotlin
class SyncEngine {
    // 替代 broadcast channel：UI 自动响应最新快照
    val globalState: StateFlow<SyncGlobalState>

    // 替代 emit_folder_content_changed：UI collect 后刷新列表
    val folderContentChanged: SharedFlow<Unit>

    // 替代 transfer_update_tx
    val transferUpdates: SharedFlow<Unit>

    // 替代 upload_failed（前端 toast）
    val uploadFailed: SharedFlow<UploadFailedEvent>

    // 替代 folder_sync_progress
    val folderSyncProgress: StateFlow<FolderSyncProgress?>
}
```

- `StateFlow<SyncGlobalState>` 自带「最新值」语义，新订阅者立即拿到当前状态，**天然解决「先订阅再 start」**问题。
- `revision` 仍保留在 `SyncGlobalState` 内，作为跨 ViewModel 的乱序防护兜底（`distinctUntilChanged` + revision 比较）。

### 9.3 传输队列乱序保护

transfer 队列用独立 `StateFlow<List<TransferTask>>`，但 `loadAll()` 的两重保护（requestId + revision CAS）仍需保留：

- Kotlin 协程并发仍可能乱序，`StateFlow` 的 `value` 赋值不是 CAS。
- 用 `update { }` 原子更新，内部按 `task.id` 做 `stateRevision` 比较后再替换条目。

```kotlin
private val _tasks = MutableStateFlow<List<TransferTask>>(emptyList())
val tasks: StateFlow<List<TransferTask>> = _tasks.asStateFlow()
private val taskRevisions = ConcurrentHashMap<Long, Long>()

suspend fun loadAll() {
    val reqId = ++lastLoadRequestId
    val fresh = api.transferListAll()
    if (reqId < lastLoadRequestId) return
    _tasks.update { current ->
        current.map { existing ->
            val incoming = fresh.firstOrNull { it.id == existing.id }
            if (incoming != null && incoming.stateRevision > (taskRevisions[existing.id] ?: -1)) {
                taskRevisions[existing.id] = incoming.stateRevision
                incoming
            } else existing
        } + fresh.filter { new -> current.none { it.id == new.id } }
    }
}
```

### 9.4 单例注入

Compose Multiplatform (CMP) 用 DI（如 Koin / Kotlin-inject）替代 `Lazy` 全局单例，作用域更清晰：

- `DriveClient` / `FilesApi` / `DB` / `StatusAggregator`：Application scope 单例。
- `MountManager` / `SyncEngine`：Session scope（与登录态绑定），登出时整个 scope 销毁，等价于 `drop_runtime_async`。
